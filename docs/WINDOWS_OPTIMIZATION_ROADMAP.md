# Windows 优化路线图

## 概述

本文档记录了 OxideTerm 在 Windows 平台上的性能优化历程和未来计划。

**目标**：让 Windows 平台上的输入响应速度达到 macOS 水平。

**现状**：Windows 输入延迟从 ~80ms 降至 <30ms（2025-01-15）

---

## 已完成的优化（P0-P2）

### ✅ P0：系统定时器精度优化

**问题**：Windows 省电模式下系统定时器精度为 15.6ms，导致所有前端优化被强制对齐到 15ms 边界。

**解决方案**：
- 在应用启动时调用 Windows API `timeBeginPeriod(1)`
- 在应用退出时调用 `timeEndPeriod(1)` 恢复系统默认值

**实施细节**：
- 文件：`src-tauri/src/lib.rs`
- 模块：`windows_timer`
- 依赖：`windows = { version = "0.52", features = ["Win32_Media_Multimedia"] }`

**效果**：
- ✅ Tokio 调度精度从 15.6ms 提升至 1ms
- ✅ WebSocket 和异步任务调度延迟显著降低
- ✅ 整体响应性提升 10-20ms
- ⚠️ 系统电源消耗增加约 1-3%（可接受）

**验证方法**：
```rust
// 查看日志是否显示
✅ Windows high-precision timer enabled (1ms precision)
```

---

### ✅ P1：禁用 Windows RAF 批处理

**问题**：`requestAnimationFrame` 在 Windows WebView2 下的同步开销非常大，导致最多 16ms 的额外延迟。

**解决方案**：
- 创建平台检测工具 `src/lib/platform.ts`
- 在 Windows 上直接写入终端数据，跳过 RAF 批处理
- macOS/Linux 继续使用 RAF 批处理以提升性能

**实施细节**：
- 文件：`src/components/terminal/TerminalView.tsx`
- 逻辑：`if (platform.isWindows) 直接写入 else RAF 批处理`

**效果**：
- ✅ Windows 输入延迟减少 10-16ms
- ✅ macOS/Linux 性能保持不变
- ✅ 代码简洁，易维护

**注意事项**：
- ⚠️ 需要配合 IME 处理（P2）

---

### ✅ P2：IME 输入法合成兼容

**问题**：禁用 RAF 后，Windows 的中文/日文输入法候选框会因为渲染太快而定位出错，导致候选框在屏幕左上角乱跳。

**解决方案**：
- 监听 `compositionstart` 和 `compositionend` 事件
- 在 IME 合成期间恢复 RAF 批处理
- 非合成期间使用直接写入

**实施细节**：
- 文件：`src/components/terminal/TerminalView.tsx`
- 状态：`isComposingRef`
- 事件监听：`addEventListener('compositionstart')`

**效果**：
- ✅ 解决 Windows 中文/日文输入法候选框定位抖动问题
- ✅ 非输入法场景下享受极致响应速度
- ✅ 输入法场景下自动切换到稳定模式

**测试方法**：
1. 打开 Windows 中文输入法（微软拼音）
2. 在终端中输入中文，观察候选框是否跟随光标
3. 切换到英文输入，观察是否保持快速响应

## 未来优化计划（P3-P4）

### 🟣 P3：高级优化（可选）

这些优化可以进一步提升 Windows 体验，但需要更多实施和测试工作。

#### 3.1 本地回显（Local Echo）

**目标**：从感官上将输入延迟降至接近 0ms

**原理**：
- 用户输入字符时，前端立即在光标处渲染一个半透明"影子"字符
- 不等待 WebSocket 返回
- 当后端真正的确认字符传回时，替换掉这个"影子"

**实施复杂度**：高

**预期效果**：
- ✅ 用户感觉输入"瞬间响应"
- ✅ 延迟感降至接近 0ms
- ⚠️ 需要处理回显确认机制
- ⚠️ 可能增加前端复杂度

**实施步骤**：
1. 在 `TerminalView.tsx` 中添加本地回显映射
2. 对可打印字符生成本地回显标记
3. 在 WebSocket 接收处理中检测服务器确认序列
4. 移除已确认的本地回显

**注意事项**：
- 需要与后端协议配合（添加确认序列）
- 控制字符不应回显
- 需要防止内存泄漏（100ms 后清理未确认的回显）

---

#### 3.2 背压监测和数据丢弃策略

**目标**：避免网络卡顿恢复后的"快进"跳动

**问题**：增大 WebSocket 缓冲区后，如果网络卡顿，数据会堆积并在恢复时"快进"跳动。

**解决方案**：
- 实现背压监测（追踪待处理数据大小）
- 设置阈值（如 50KB）
- 超过阈值时丢弃中间帧，直接跳转到最新状态

**实施复杂度**：中

**预期效果**：
- ✅ 避免网络卡顿后的数据堆积
- ✅ 恢复后不会出现"快进"现象
- ✅ 用户体验更流畅

**实施步骤**：
1. 添加 `pendingSizeRef` 追踪数据大小
2. 在 `MSG_TYPE_DATA` 处理中更新计数
3. 检查是否超过阈值
4. 超过时清空缓冲区，直接写入最新数据

**注意事项**：
- 阈值需要根据实际使用情况调优
- 丢弃数据可能导致终端内容不一致
- 需要在开发环境中记录丢弃事件

---

#### 3.3 TCP_NODELAY 验证和增强

**目标**：确保网络栈不增加额外延迟

**现状**：已设置 TCP_NODELAY，但没有验证是否成功

**优化方向**：
1. 验证 TCP_NODELAY 是否真的生效
2. 在 Windows 上使用优化的 Socket 创建流程
3. 设置合适的发送和接收缓冲区大小

**实施复杂度**：低

**预期效果**：
- ✅ 确认 TCP_NODELAY 生效
- ✅ 减少网络延迟 5-10ms
- ✅ 提升网络吞吐量

**实施步骤**：
1. 在设置后验证 `stream.nodelay()` 返回值
2. 添加日志记录验证结果
3. 使用 `TcpSocket::new_v4()` 手动创建 Socket
4. 设置 `set_send_buffer_size(65536)` 和 `set_recv_buffer_size(65536)`

**注意事项**：
- Windows 和 Unix 的 Socket API 不同
- 需要条件编译 `#[cfg(target_os = "windows")]`

---

#### 3.4 WebSocket 缓冲区调整

**目标**：减少 Windows 上的背压和阻塞

**现状**：所有平台使用统一的 4096 容量

**优化方向**：
- Windows 使用更大的缓冲区（16384）
- macOS/Linux 保持 4096

**实施复杂度**：低

**预期效果**：
- ✅ 减少 Windows 上的背压
- ✅ 避免阻塞
- ⚠️ 增加内存使用（约 12KB）

**实施步骤**：
1. 定义平台特定的常量
2. 替换所有固定的 4096 为 `FRAME_CHANNEL_CAPACITY`
3. 在 Windows 上使用 16384

**注意事项**：
- 需要在所有创建 channel 的地方修改
- 需要测试内存使用情况

---

#### 3.5 Canvas 渲染器优化

**目标**：可能比 WebGL 更快的渲染性能

**问题**：WebGL 在某些 Windows 配置下性能不佳

**解决方案**：
- 添加渲染器选择配置（webgl/canvas）
- Windows 默认使用 Canvas，可手动切换到 WebGL
- 提供 UI 设置让用户选择

**实施复杂度**：低

**预期效果**：
- ✅ Windows 上可能提升渲染性能 5-10ms
- ✅ 更好的兼容性
- ⚠️ 需要用户手动切换

**实施步骤**：
1. 在设置中添加 `renderer` 选项
2. 修改 WebGL 加载逻辑
3. Windows 默认使用 'canvas'
4. 添加用户设置 UI

**注意事项**：
- 需要更新设置类型定义
- 需要持久化用户选择

---

### 🔵 P4：长期优化（探索性）

这些优化需要更多研究和验证，建议在完成 P3 后根据实际需求决定是否实施。

#### 4.1 Tokio 运行时配置优化

**目标**：改善 Windows 上的任务调度延迟

**方向**：
- 增加工作线程数（从默认调整为 4）
- 使用多线程调度器（`rt-multi-thread`）
- 针对平台特性优化线程池配置

**实施复杂度**：高

**预期效果**：
- ✅ 改善调度延迟 2-5ms
- ✅ 更好的多核利用率
- ⚠️ 可能增加 CPU 使用
- ⚠️ 需要仔细调优

---

#### 4.2 输入批处理优化

**目标**：减少 Windows 上的 WebSocket 发送开销

**原理**：
- 快速连续输入时，批量发送字符
- 设置 8-16ms 的延迟窗口
- 减少网络往返次数

**实施复杂度**：中

**预期效果**：
- ✅ 减少 WebSocket 开销
- ⚠️ 可能增加延迟感
- ⚠️ 需要仔细调优延迟窗口

---

#### 4.3 内存分配优化

**目标**：减少 GC 压力和内存分配开销

**方向**：
- 预分配固定大小的 buffer
- 复用 buffer 而非每次创建新数组
- 减少对象创建

**实施复杂度**：中

**预期效果**：
- ✅ 减少内存分配
- ✅ 降低 GC 压力
- ✅ 提升整体性能

## 测试计划

### 回归测试

每次优化后必须进行：

1. **macOS/Linux 测试**
   - 确保性能没有退化
   - 验证所有终端功能正常
   - 检查输入延迟 <30ms

2. **Windows 功能测试**
   - 快速输入（打字、代码编辑）
   - 中文/日文输入法
   - 复制粘贴
   - 特殊字符
   - 滚动和搜索

3. **压力测试**
   - 快速输入大量字符
   - 同时运行多个终端会话
   - 长时间使用后的性能表现

### 性能对比测试

1. **主观测试**
   - 感受响应速度
   - 对比优化前后

2. **客观测试（可选）**
   - 屏幕录制（60fps+）
   - 逐帧分析按键到显示的时间
   - 对比 macOS 和 Windows 的延迟

## 已知限制和权衡

### 系统定时器优化

**限制**：
- 增加系统电源消耗约 1-3%
- 影响整个系统的定时器精度

**权衡**：
- ✅ 用户体验提升远大于电源成本
- ✅ 对桌面应用影响可忽略
- ✅ 应用退出时自动恢复

### RAF 禁用

**限制**：
- 必须配合 IME 处理
- 可能增加渲染开销

**权衡**：
- ✅ 延迟降低 10-16ms
- ✅ 仅影响 Windows
- ✅ macOS/Linux 保持不变

### IME 处理

**限制**：
- 增加代码复杂度
- 需要维护状态

**权衡**：
- ✅ 保证输入法稳定
- ✅ 自动切换策略
- ✅ 对非输入法用户无影响
